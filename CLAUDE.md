## Project Overview

**esp-fc** is an open-source flight controller firmware for ESP32/ESP8266/RP2040 boards, compatible with the Betaflight ecosystem (MSP protocol, Betaflight Configurator). It targets hobbyist quadcopter drones.

## Build System

Uses **PlatformIO** with Arduino framework. Supported targets: `esp32`, `esp32s2`, `esp32s3`, `esp32c3`, `esp8266`, `rp2040`, `rp2350`.

```bash
# Build specific target
pio run -e esp32

# Build and upload to device
pio run -e esp32 -t upload

# Run unit tests (native host build)
pio test -e native

# Run a single test suite
pio test -e native -f test_fc

# Docker alternative (no local PlatformIO install required)
docker-compose run pio pio run -e esp32
docker-compose run pio pio test -e native
```

Build flags of note (uncomment in `platformio.ini` to activate):
- `-DESPFC_DEBUG_SERIAL` — enables serial debug output
- `-DESPFC_DEV_PRESET_DSHOT` / `-DESPFC_DEV_PRESET_BRUSHED` — dev presets for motor protocols
- `-DESPFC_DEV_PRESET_BLACKBOX_SERIAL=1` — enables blackbox over serial

## Architecture

### Execution Model

**Single-core** (ESP8266, RP2040): `setup()` → `espfc.begin()`, then `loop()` → `espfc.update()` + `espfc.updateOther()`.

**Dual-core ESP32 (FreeRTOS)**:
- **gyroTask** (core 1, high priority): 4kHz hardware-timer-driven, runs `espfc.update(true)` — gyro read, IMU fusion, PID
- **pidTask** (core 0, lower priority): runs `espfc.updateOther()` — telemetry, RC input, motor arming, CLI/MSP

### Core Components (`lib/Espfc/src/`)

| Component | Purpose |
|-----------|---------|
| `Espfc.h/cpp` | Top-level orchestrator — owns all subsystems |
| `Model.h` | Shared mutable state + EEPROM-backed configuration |
| `Hardware.h` | Platform hardware init (pins, buses) |
| `Control/Controller.h` | Main flight control loop (rates → PIDs → mixer) |
| `Control/Fusion.h` | AHRS (Madgwick/Mahony/Kalman) attitude estimation |
| `Control/Altitude.hpp` | Altitude hold using barometer |
| `Device/` | Hardware drivers (gyros, baros, mags, receivers, buses) |
| `Sensor/` | Sensor managers that wrap Device drivers |
| `Output/Mixer.h` | Motor mixing (throttle + PID outputs → motor commands) |
| `Connect/Msp.h` | MSP protocol — Betaflight Configurator communication |
| `Connect/Cli.h` | Serial CLI for configuration |
| `Blackbox/` | Flight data logging (serial or SPI flash) |
| `Telemetry/` | CRSF and text telemetry output |
| `Utils/Filter.h` | Digital filters: LPF, biquad, dynamic notch, RPM notch |
| `Utils/Timer.h` | Rate-based task scheduling |

### External Libraries (`lib/`)

- **AHRS/** — Madgwick, Mahony, Kalman fusion algorithms
- **EscDriver/** — PWM/OneShot/DSHOT ESC protocol drivers (platform-specific: ESP32, C3, RP2040)
- **EspWire/** — I2C/SPI wrapper
- **betaflight/** — Betaflight-sourced sensor drivers and filter implementations
- **Gps/** — GPS parsing

### Data Flow

```
RC Input (SBUS/IBUS/CRSF/PPM/ESP-NOW)
    → Input → RcCommand
    → Controller (rates) → PID → setpoint
    → Fusion (gyro + accel + mag → attitude)
    → Mixer → motor commands
    → EscDriver → ESC output (PWM/DSHOT)
```

`Model` is the central shared state accessed by all components. Configuration is persisted via `Utils/Storage.h` (EEPROM).

## Testing

Tests live in `test/` and use **Unity** + **ArduinoFake** (mocking Arduino APIs for host builds).

```
test/
├── test_fc/         # Flight controller core tests
├── test_esc/        # ESC driver tests
├── test_input_crsf/ # CRSF protocol parsing tests
├── test_msp/        # MSP protocol tests
└── test_math/       # Math/filter utility tests
```

The `native` environment (`-DUNIT_TEST`, `-DNO_GLOBAL_INSTANCES`) compiles for the host and disables hardware-specific globals.

## Key Conventions

- **C++17** throughout (`-std=gnu++17`)
- Platform guards use `#ifdef UNIT_TEST`, `#ifdef ESP32`, `#ifdef ARCH_RP2040`, etc.
- `IRAM_ATTR` marks time-critical ISR/gyroTask functions; defined away (`""`) on native/RP2040
- Device drivers follow a pattern: `begin()` for init, `update()` for periodic reads, returning status codes
- The `Model` struct is passed by reference into most components — avoid adding global state elsewhere
- Timing-critical gyro path must remain lock-free; use `SemaphoreHandle_t` or atomic flags for cross-core communication

## CI/CD

GitHub Actions (`.github/workflows/platformio.yml`) runs `pio test -e native` first, then builds all 7 targets if tests pass. Firmware artifacts are uploaded as release assets. Merged firmware images (`firmware_0x00.bin`) are generated by `merge_firmware.py` for ESP32 variants.
